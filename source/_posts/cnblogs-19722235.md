<h1 id="关于内存取证">关于内存取证</h1>
<h3 id="什么是内存镜像vmem">什么是内存镜像（.vmem）？</h3>
<p><code>.vmem</code> 是 VMware 虚拟机运行时的内存快照文件，相当于把电脑 RAM 里的所有内容完整保存成一个文件。通过分析它，我们可以还原出案发时系统的几乎所有状态：运行的程序、网络连接、注册表、甚至密码。</p>
<h3 id="内存取证工具volatility">内存取证工具Volatility</h3>
<p>Volatility 是内存取证领域最主流的开源工具，能从内存镜像中提取各类信息。</p>
<p>使用 Volatility 3（命令为 <code>vol.exe</code>）即可</p>
<h1 id="otterctf-2018what-the-password">[OtterCTF 2018]What the password?</h1>
<p>:::warning<br>
你获取了 Rick 的电脑内存镜像，能否从中提取出他的用户密码？</p>
<p>flag格式：<code>NSSCTF{Rick的密码}</code></p>
<p>:::</p>
<p>Windows 系统若配置了自动登录，密码会以明文存储在 LSA Secrets（<code>DefaultPassword</code>）中，可直接从内存提取。</p>

<h3 id="step3获取-pc-名">Step3:获取 PC 名</h3>
<p>Windows 的计算机名存储在注册表的固定路径下：</p>
<pre><code class="language-sql">HKEY_LOCAL_MACHINE\SYSTEM\ControlSet001\Control\ComputerName\ComputerName
</code></pre>
<p>Windows 注册表是一个层级数据库，存储系统和软件的几乎所有配置。</p>
<p><code>ControlSet001</code> 是活跃的控制集（类似"当前配置"）</p>
<p>:::color1<br>
Windows 的注册表（Registry）是一个树形数据库，<code>HKEY_LOCAL_MACHINE\SYSTEM</code> 分支下保存着操作系统级别的配置。计算机名这类"机器身份"信息由系统在安装时写入，后续可以通过"系统属性"界面修改，修改后依然写回到这个注册表路径。</p>
<p>Volatility 的 <code>printkey</code> 插件直接从内存中解析注册表蜂巢（hive）文件的内存映射，因此即使没有硬盘镜像，只要内存里有注册表的缓存，就能读出这些值。</p>
<p>:::</p>
<p>Volatility 3 的 <code>windows.registry.printkey</code> 插件可以直接读取注册表内容：</p>

<p><code>Data</code>是<code>WIN-LO6FAF3DTFE</code>注册表<code>ComputerName</code>的键值，就是PC的名字</p>
<p>第一行的 <code>(Default)</code> 是默认值，<code>mnmsrvc</code> 是远程桌面服务的服务名，不是计算机名，忽略它</p>
<p>所以PC 名就是<code>WIN-LO6FAF3DTFE</code></p>
<p>:::color5<br>
<code>NSSCTF{WIN-LO6FAF3DTFE-192.168.202.131}</code></p>

<h3 id="为什么账号名会明文存在内存里">为什么账号名会明文存在内存里？</h3>
<p>游戏客户端在登录后需要实时显示账号名（界面 UI、角色选择页面等），因此账号名会以明文字符串形式存在进程的堆内存（heap）或数据段中。除非游戏客户端专门做了内存保护（如加密存储），否则只要进程在运行，账号名就可以被读出来。这也是游戏外挂/内存编辑工具能读取游戏数据的根本原因。</p>
<h3 id="什么是进程内存dump">什么是进程内存Dump？</h3>
<p>Volatility 可以把某个进程占用的全部虚拟内存导出成一个文件（<code>.dmp</code>），之后就可以像普通文件一样用 Python、strings 等工具搜索里面的内容，不需要反复调用 Volatility。</p>

<h1 id="otterctf-2018silly-rick">[OtterCTF 2018]Silly Rick</h1>
<p>:::warning<br>
傻乎乎的Rick总是忘记自己邮箱的密码，所以他使用了一个在线的“已保存密码服务”（Stored Password Services）来存储密码。他每次都会复制粘贴这个密码，以免输错。<br>
Rick 的邮箱密码是什么？</p>
<p>flag格式:<code>NSSCTF{邮箱密码}</code></p>
<p>:::</p>
<ol>
<li>Rick 用在线密码存储服务保存密码</li>
<li>他每次都复制粘贴密码，不手打</li>
</ol>
<p>"复制粘贴"意味着密码此刻就在 Windows 剪贴板里。</p>
<p>内存快照捕获时，剪贴板内容还残留在内存中，Volatility 可以直接读出来。</p>
<pre><code class="language-plain">用户按 Ctrl+C
      ↓
程序调用 SetClipboardData(CF_UNICODETEXT, 密码字符串)
      ↓
win32k.sys 在内核堆中分配内存块，存储 UTF-16LE 编码的文本
      ↓
内存快照时数据仍在 ← Volatility 2 的 clipboard 插件可直接提取
</code></pre>

<h3 id="vol2-clipborad插件的原理">vol2 clipborad插件的原理</h3>
<ol>
<li>找到 win32k.sys 在内核中的基地址</li>
<li>定位 tagWINDOWSTATION 结构（记录了所有窗口站）</li>
<li>遍历每个窗口站的 tagCLIPDATA 链表</li>
<li>对每个 CF_UNICODETEXT 条目，读取 cbData 字节的数据</li>
<li>以 UTF-16LE 解码，输出明文</li>
</ol>
<h3 id="为什么剪贴板内容在内存里">为什么剪贴板内容在内存里？</h3>
<p>Windows 剪贴板不会自动清除，<strong>只有以下情况才会清空</strong>：</p>
<ol>
<li>新内容写入剪贴板（下一次 Ctrl+C）</li>
<li>程序主动调用 <code>EmptyClipboard()</code></li>
<li>系统重启</li>
</ol>
<p>内存快照时Rick可能刚复制完还没粘贴或者粘贴完程序没有清除剪贴板。</p>
<p>这样这样Windows剪切板数据在有新内容覆盖前始终保留在内存中。</p>
<p>因为内存快照捕获的是某一瞬间的内存状态。所以，时机合适就能捕获到剪切板内容</p>
<p>还有还有。运行 Vol2 时可能会看到类似这样的警告：</p>
<pre><code class="language-plain">*** Failed to import volatility.plugins.malware.apihooks (NameError: name 'distorm3' is not defined)
</code></pre>
<p>这是因为我这里<code>distorm3</code>（反汇编库）没有安装，只影响 <code>apihooks</code>、<code>ssdt</code> 等需要反汇编的插件，<strong>不影响 clipboard、pstree、netscan 等大多数常用插件</strong>，可以忽略。当然如果安了就不会报错了。</p>
